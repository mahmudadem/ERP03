# Internationalization and RTL hardening

ERP03 uses `i18next` / `react-i18next` with locale JSON files under:

- `frontend/src/locales/en/`
- `frontend/src/locales/ar/`
- `frontend/src/locales/tr/`

## Production QA 278k

Telegram QA screenshots showed Arabic sessions with English text still visible in:

- Sales dashboard header, KPI cards, recent invoice table, top-client card, quick navigation, and recent activity labels.
- Purchase Settings default-account section for AP sub-account generation.

The fix keeps the existing translation architecture:

- React pages use `useTranslation(...)` and `t(key, fallback)` for user-facing strings.
- Sales dashboard keys live under `common:sales.home.*`.
- Purchase AP-generation keys live under `purchases:settings.*`.
- AP account-code preset labels are stored as translation keys instead of hardcoded module constants.
- Shared DatePicker quick-select labels live under `common:datePicker.*` and must stay translated because
  the picker is reused by reports across modules.

## Rules for future fixes

- Do not add hardcoded English for visible UI labels, table headings, button text, help text, or modal titles.
- Add keys to all active locale files: English, Arabic, and Turkish.
- Keep accounting/control terminology consistent:
  - AR = receivables / الذمم المدينة.
  - AP = payables / الذمم الدائنة.
  - Parent account = الحساب الأب.
- If a page is in a non-`common` namespace, verify the JSON file for that namespace exists in all three locale folders.
- Validate with JSON parse plus frontend typecheck/build after editing locale files.
