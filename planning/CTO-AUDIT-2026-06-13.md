## 1. Executive summary

**You are not missing features. You've built the hard 80% of a real ERP — the accounting engine is genuinely solid — but the project has been stuck in "build and polish" mode for weeks while the things that turn code into a product (deployment, automated checks, a real pilot user) sit at zero.**

The overwhelm exists because the work now generates more work instead of converging. The fix is not another feature; it's declaring v1 frozen and shipping a pilot.

## 3. Why the owner is overwhelmed — diagnosis

1. **The QA bottleneck.** QA-QUEUE.md holds ~40 untested features. Agents produce work faster than one human can verify it, and there are **zero frontend tests and zero E2E tests**, so every UI change needs the owner's eyes. The project's own quality bar ("every feature works end-to-end before moving on") is being structurally violated.
2. **Drift into unbounded polish.** The last ~3 weeks of journal entries are almost entirely cosmetic (table fonts, row colors, header densities, hub redesigns; tasks ~186–216). Polish is comfortable work and never ends. Meanwhile the standing priority (Apex shell cutover QA, estimated 2–4 hours) has been "next" since June 5.
3. **Structural debt accumulating quietly.** The working branch is **149 commits ahead of `main`** with a misleading name, ~20 stale branches exist, and there is **no CI**.
4. **No finish line defined.** The system has **never been deployed** — no hosting config in firebase.json, no monitoring, no backups, everything on local emulators. The vision says "validate the idea then hand off to engineers" — validation requires a real company using the system, and that milestone has no date.

## 8. Database choice — Firebase for an ERP? (added 2026-06-13 on owner's request)

**Short answer: the concern is valid, the chosen approach (Firestore now, SQL later) is the right one — but the escape hatch being counted on is quietly rusting.**

### Why Firestore is an unusual choice for an ERP

Double-entry accounting is the most relational workload in software. Firestore's specific pain points for this product:

- **No joins, no GROUP BY:** trial balance, P&L, aging, and statements are aggregations over many rows. Today the code fetches arrays and reduces them client-side — already flagged in the project's own backlog as a scalability rabbit hole.
- **Cost model:** Firestore bills per document read. Every report run re-reads thousands of ledger documents; on SQL the same query is essentially free on a $25/month instance.
- **No database-level integrity:** no foreign keys, no unique constraints. All integrity is application-level (well built here, but the DB catches nothing the app misses).
- **Write contention:** sequential voucher numbering relies on counter documents (~1 sustained write/sec per doc) — a classic Firestore hotspot.
- **Auditor/BI expectations:** accountants and auditors expect SQL exports and BI connectivity; Firestore requires BigQuery pipelines for that.

### Why the choice was still right for this phase

- For a solo, non-technical founder, Firebase eliminated an entire operational discipline (DBA, backups, scaling, connection pooling) — plausibly months of survival time.
- At pilot scale (1–3 companies), none of the limits above bite, and the cost is ~zero.
- The architecture hedged properly — far beyond the typical "we'll migrate later" claim: repository pattern enforced by boundary tests, a **2,207-line Prisma schema with 95 models**, **82 Prisma repository implementations**, and a real `DB_TYPE=FIRESTORE|SQL` switch in the DI container.

### New audit findings on the SQL path

1. **The SQL path has never been tested.** The project's own switch guide states: "Infrastructure Complete, **Untested** — never run against a real database; zero SQL integration tests." An untested migration path is a hope, not a plan.
2. **It is drifting.** Measured directly: **110 Firestore repository classes vs 82 Prisma — 29 have no SQL counterpart.** The missing list is precisely the last two months of features: PostingLog, IdempotencyKey, RecordChangeLog (audit), PeriodLockOverride, Quote, PriceList, PromotionRule, Salesperson, RecurringInvoiceTemplate, Customer/Vendor Groups, PurchasePriceList, CommissionEntry, CreditOverride, CommunicationsSettings, FieldLibrary, plus all 13 AI repositories. Every week of new features widens the gap.
3. **The switch guide is stale** (last updated 2026-04-20; says "74 models / 85 repos" — reality has moved on).

### Ruling

1. **Do NOT migrate now.** It would torpedo the ship-a-pilot plan, and at pilot scale Firestore is fine.
2. **Put the SQL path under drift control (~1 day):** an architecture test that fails CI when a Firestore repository has no Prisma counterpart, with an explicit allowlist for intentionally Firestore-only areas. Add this to Phase 1 (CI week) of the plan.
3. **Decide exemptions explicitly (clarified 2026-06-13 after owner challenge):** exempt the 13 AI repositories from drift control *for now only* — they need no SQL mirrors while Firestore is the production database. The end state is **strict one-database**: when migration happens, the AI repositories get their Prisma mirrors too (mechanical CRUD work, days not weeks) and ALL data moves to SQL. Old chat history is disposable conversation data, not business records — archive or discard it at migration rather than porting it. A permanent hybrid (business data in SQL, chats in Firestore) is a legitimate industry pattern ("polyglot persistence") but means operating two data stores forever; rejected here in favor of handoff simplicity.
4. **Set explicit migration triggers** instead of "someday": the natural moment is the handoff to professional engineers (they will want Postgres), or the first customer whose reporting volume/cost hurts — whichever comes first. Managed Postgres (Supabase / Cloud SQL) preserves the no-ops advantage, so migrating does not mean losing what made Firebase attractive. Firebase Auth and FCM stay either way (already designed that way).
